## 核心结论

> [Evidence] 本仓库**没有通用插件系统**（无插件 API、无动态插件加载、无插件市场运行时）。但存在若干**受控的扩展点**：配置驱动的类覆盖（`OPENHANDS_CONFIG_CLS` + `get_impl`）、FastAPI 路由叠加、FastMCP 工具注册、技能（markdown）、`Injector[T]` DI。
> 不要把这些扩展点夸大为"插件系统"。

## 扩展点 1：配置驱动的类覆盖（`OPENHANDS_CONFIG_CLS` + `get_impl`）

- 接口/类型：`ServerConfig`（`server_config.py:9-56`）的类属性字符串；`get_impl(cls, qualname)`（`import_utils.py:43-78`）。
- 注册方式：设环境变量 `OPENHANDS_CONFIG_CLS=<全限定类名>`；`load_server_config()` 调 `get_impl(ServerConfig, ...)` 动态导入并实例化。
- 加载时机：`shared.py:14` import 时即解析。
- 生命周期：进程级单例。
- 可访问资源：可覆盖 `settings_store_class`、`secret_store_class`、`user_auth_class`、`conversation_secret_enricher_class`、`analytics_user_provider_class`。
- 权限边界：无显式沙箱（同进程动态导入，信任部署者）。
- 调用路径：`load_server_config → get_impl → 实例化 → .verify_config()`。
- 示例实现：`enterprise/server/config.py:56` `SaaSServerConfig`。
- 稳定性：[Evidence] 但 lifespan 选择用**字符串嗅探** `'saas' in OPENHANDS_CONFIG_CLS.lower()`（`config.py:174-183`）而非 `get_impl`，**不一致且脆弱**。
- 是否满足扩展判据：✅ 可加新实现 ✅ 入口会发现 ✅ 调用方依赖抽象（`UserAuth` ABC 等）⚠️ 错误隔离弱（动态导入异常即启动失败）。
- Evidence：`server_config.py:48-56`、`import_utils.py:43-78`、`enterprise/server/config.py:56`。

## 已确认事实

- 无通用插件系统。
- 真扩展点：类覆盖、路由叠加、MCP 工具、技能、DI、后端策略。
- Agent/Tool 级扩展在外部 SDK。

## 合理推断

- 类覆盖 + 路由叠加是 enterprise "superset 部署"的核心机制。

## Unknown 与待验证事项

- [Unknown] 外部 SDK 的 hook/condenser/custom-agent 扩展 API。

## 批评性评估

- lifespan 字符串嗅探与 `get_impl` 模式不一致。
- 路由覆盖用 monkey-patch，非显式扩展契约。
- MCP 工具硬编码，无运行时注册。

## 建设性改善建议

- [Recommendation] lifespan 选择改用显式字段 + `get_impl`，消除字符串嗅探。优先级：中。
- [Recommendation] 路由覆盖提供显式 `override_router` 契约。优先级：低。
- [Recommendation] 若需要第三方扩展，设计带权限隔离的插件 API；否则明确文档"无插件系统"。优先级：低。
